# docx troubleshooting

## `set_text` cannot find the desired paragraph

docx heading/paragraph indexes are based on **document order**. Use the `index` returned by `inspect_doc.py`. Headings and paragraphs are **separate sequences** (heading:0, heading:1, … + paragraph:0, paragraph:1, …).

## Spec ops fail with "no .spec.json found"

If the docx was generated by `build_doc.py`, `<file>.docx.spec.json` exists next to it automatically. For documents from other tools, first run:

```
extract_doc.py --in foo.docx --out foo.docx.spec.json
edit_doc.py --in foo.docx --patch p.json --out out.docx
```

## TOC does not appear in the body

The `toc` block only inserts Word field code. Open the file in Word/LibreOffice and press F9 (refresh) to populate actual entries. python-docx does not evaluate fields.

## Image size stays the same after swap

`swap_image` only replaces bytes. Original display size is preserved — the new image's actual resolution is ignored. To change size too, use a spec op to `replace_block` the image block and set a new width_in.

## Code block font changes to another monospace

docx stores only font names. If the viewer lacks Menlo/Consolas, it substitutes. Use a common font in the theme, such as `mono_font: "Courier New"`.

## Korean renders as boxes

The viewer may lack East-Asian fonts, or the theme may force an ASCII-only font. Override `body_font` in the theme to Apple SD Gothic Neo / Malgun Gothic.

## Table style does not render as expected

`style: "grid"` is consistent in most viewers. `"light"` and `"plain"` depend on Word built-in style names — if missing, they fall back to the default style. If the docx source requires a specific style, it should be embedded as a custom style in a docx template and connected via `theme_overrides`, but that is currently out of scope.
